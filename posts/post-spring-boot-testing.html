How I Added Unit and Integration Tests to My Spring Boot App — A Junior Dev's Honest Guide

JUnit 5, Mockito, MockMvc, Spring Security Test, and @DataJpaTest — what each one does and why I used it

By Keshav Callychurn

I built a full-stack birthday reminder app called RememberBirthdays using Spring Boot, Keycloak for authentication, PostgreSQL, and React. It's been deployed on AWS and works well — but for a long time it had exactly one test: the auto-generated contextLoads() that ships with every Spring Boot project.

Not great for a portfolio. So I spent time properly adding tests across all three layers of the application. This article walks through what I built, the technologies I used, and the challenges I ran into along the way.

The Architecture

The backend is a layered Spring Boot REST API:

  • Controller layer — handles HTTP requests and security rules
  • Service layer — contains business logic
  • Repository layer — talks to PostgreSQL via Spring Data JPA

Each layer needed a different type of test.

The Technologies

  • JUnit 5 — the test runner. Every test method is annotated with @Test.
  • Mockito — the mocking framework. Replaces real dependencies with fakes.
  • Spring Boot Test / MockMvc — for testing the web layer including HTTP status codes and JSON responses.
  • Spring Security Test — provides the jwt() helper for testing secured endpoints.
  • @DataJpaTest with H2 — runs repository tests against an in-memory database.
1Layer 1 — Service Unit Tests

Service tests are pure unit tests. No Spring context is loaded, so they run in milliseconds.

The pattern is simple: mock every dependency with @Mock, inject them into the real class with @InjectMocks, and then call the method you want to test.

@ExtendWith(MockitoExtension.class)
class UserServiceTest {
    @Mock UserRepository userRepository;
    @InjectMocks UserService userService;
    // ...
}

One challenge I hit: PersonService reads the current logged-in user from SecurityContextHolder — a static global. In a unit test there is no HTTP request, so SecurityContextHolder is empty and the service crashes immediately.

Fix: Manually place a fake JwtAuthenticationToken into the SecurityContextHolder before each test and clear it in @AfterEach.

This taught me that testing code that depends on static state requires you to manage that state yourself.

2Layer 2 — Controller Slice Tests

Controller tests use @WebMvcTest, which loads only the web layer — controllers, security config, and filters. No services or repositories are loaded (you @MockBean them).

The biggest challenge here was Keycloak. My SecurityConfig declares a JwtDecoder bean that connects to the Keycloak JWKS endpoint on startup. In a test environment there is no Keycloak server, so the context would crash on startup.

Two things were needed:

  • @MockBean JwtDecoder — overrides the real bean so Spring does not try to connect to Keycloak
  • jwt() from Spring Security Test — injects a fake authentication into the security context for each request

With these in place I could test real security behaviour including:

  • Unauthenticated requests returning 401
  • A non-owner getting 403 when trying to delete someone else's record
  • An admin bypassing the ownership check and getting 200
This was the most valuable part of the whole exercise. Before writing these tests I thought my security logic worked because I had manually tested it. The tests forced me to verify every edge case systematically.
3Layer 3 — Repository Slice Tests

Repository tests use @DataJpaTest, which automatically replaces your PostgreSQL datasource with H2 in-memory and creates all your JPA entity tables from your @Entity classes.

I used TestEntityManager to insert test data directly into H2, bypassing the repository under test. This separation is important — if you use the repository to both set up and test, you cannot be sure where a failure comes from.

The tests verify that findAllByUser() correctly returns only the persons belonging to a specific user — which is a core piece of logic that ensures users can only see their own data.

Key Lessons

  • Different layers need different test types. Reaching for @SpringBootTest on everything is slow and defeats the purpose.
  • Writing tests for security logic is where you find the bugs. I found an edge case in my ownership check that I had never manually tested.
  • Mockito's verify() is just as important as assertions. Checking that save() is never called when a user already exists is a meaningful test.
  • The SecurityContextHolder is a gotcha for anyone testing Spring Security without a full context. Managing it manually is the right approach at the unit test level.

Conclusion

18 tests across 5 files. Every test meaningful, every test green. If you're building a Spring Boot portfolio project and have only the auto-generated contextLoads() test — I hope this gives you a clear path forward.
