Migrating a Full-Stack App to AWS: What Tutorials Don't Teach You

By Keshav Callychurn

After completing my AWS Cloud Practitioner certification, I wanted to go beyond multiple-choice questions and deploy something real. This is the story of migrating RememberBirthdays — a birthday reminder app — from Netlify and Railway to AWS, and every problem I hit along the way.

What I Built

RememberBirthdays lets you store birthdays and receive reminders so you never forget the people who matter. The stack:

  • Frontend: React + TypeScript
  • Backend: Java Spring Boot
  • Auth: Keycloak (OAuth2 / OIDC)
  • Database: PostgreSQL
  • Containerisation: Docker Compose

Previously deployed on Netlify (frontend) and Railway (backend + Keycloak). The goal was to migrate everything to AWS and understand what real cloud infrastructure looks like.

Architecture

The final architecture has three layers: a public CloudFront edge, a compute layer in EC2 running Docker, and a managed database in a private subnet.

User (Browser) | HTTPS v CloudFront ×3 [Frontend CDN] [Auth Proxy] [API Proxy] | | | v v v S3 Bucket EC2 t3.small (Docker Compose) (React build) +---------------------------+ Private / OAC | Keycloak <-> Spring | Public Subnet | (OAuth2) Boot API | +----------------|----------+ | JDBC v RDS PostgreSQL Private Subnet db.t3.micro app_db | keycloak_db

Key Architectural Decisions

RDS over Docker PostgreSQL

Running Postgres in a container on EC2 is cheaper, but RDS gives automated backups and managed patching. It also demonstrates understanding of managed cloud services, which is how production systems are built.

No load balancer

With fewer than 5 users per month, an ALB adds ~$16/month for zero benefit. I know exactly where the scaling bottleneck is and what the upgrade path looks like.

Kept Keycloak, didn't switch to Cognito

Keycloak adds operational overhead but it's portable — not tied to AWS. A deliberate trade-off between convenience and vendor lock-in.

EC2 in public subnet, RDS in private subnet

The EC2 needs to be reachable from CloudFront. The database should never be reachable from the internet. Only Spring Boot can connect to RDS, through an explicit security group rule.

The Problems That Actually Taught Me Something

1EC2 Ran Out of Memory on First Boot

I started with a t2.micro — 1 GB RAM. Keycloak alone needs approximately 600 MB at startup. Spring Boot needs another 300–400 MB. The containers were crashing silently with no obvious error — just services that never came up.

Fix: Resized to t3.small (2 GB RAM). Both containers started cleanly.

Understand your workload's memory requirements before choosing an instance type. Keycloak is a full identity provider, not lightweight middleware. Check container minimum requirements before you deploy.

2RDS Was Unreachable from EC2

The RDS instance existed. The JDBC connection string was correct. Spring Boot still threw connection refused on every startup.

Root cause: The RDS security group had no inbound rule allowing traffic from the EC2. In AWS, resources cannot communicate unless you explicitly configure security groups to permit it. The database was sitting in a private subnet with no path from the EC2 to reach it.

Fix: Added an inbound rule to the RDS security group: allow port 5432 (PostgreSQL) from the EC2's security group.

AWS security groups are a whitelist. Nothing talks to anything unless you explicitly allow it. This is good — defence in depth — but every connection must be consciously permitted.

3JWT Authentication Rejected Every Request

Every authenticated API call returned 401 Unauthorized. The tokens looked valid. Keycloak was running. The error was JwtDecoderInitializationException.

Root cause — two separate but related problems:

First, Keycloak was issuing tokens with an issuer claim (iss) pointing to its public EC2 URL. Spring Boot validates tokens by fetching Keycloak's public keys from an issuer URI. The iss claim in the token and the configured issuer URI have to match exactly — they didn't.

Second, from inside a Docker container on EC2, the host's public IP address often isn't routable back to the host. Spring Boot couldn't even reach Keycloak at the public IP to fetch its public keys.

Fix: A custom JwtDecoder that fetches JWKS keys from Keycloak's internal Docker hostname (always reachable) while skipping issuer URI validation entirely. Token signatures are still verified — we're not skipping security, just the URL matching check.
@Value("${keycloak.jwks-uri:http://keycloak:8080/realms/remember-birthdays/protocol/openid-connect/certs}")
private String jwksUri;

@Bean
public JwtDecoder jwtDecoder() {
    // Fetches JWKS from internal Docker hostname — always reachable
    // withJwkSetUri() validates signature but skips issuer claim check
    return NimbusJwtDecoder.withJwkSetUri(jwksUri).build();
}

The proper production fix: a domain name with HTTPS so Keycloak's internal and external hostname are identical. When KC_HOSTNAME matches the token's iss claim and is reachable both internally and externally, the mismatch disappears entirely.

4HTTPS Blocked Every API Call

The frontend loads over HTTPS via CloudFront. After login, every API call to the backend and every request to Keycloak's token endpoint failed silently.

Root cause: Mixed content. Browsers enforce a strict rule — an HTTPS page cannot make requests to HTTP resources. There is no workaround, no flag to disable it, no exception. You fix it or it doesn't work.

Fix: Created separate CloudFront distributions for Keycloak and the API. CloudFront handles HTTPS termination — the browser talks to CloudFront over HTTPS, CloudFront forwards to the EC2 over HTTP internally. The browser never sees the HTTP leg.

One non-obvious detail: CloudFront only supports HTTP origins on ports 80 and 8080. Keycloak was on port 8081. Moving Keycloak to port 80 (free since Spring Boot uses 8080) resolved this. Caching was disabled on both distributions — Keycloak and the API return dynamic responses that must never be cached.

The proper production fix: a custom domain with an ALB and ACM certificate. The ALB handles HTTPS termination cleanly, routes to both services, and eliminates the need for multiple CloudFront distributions.

5CORS Blocked API Calls from the Frontend

After fixing HTTPS, the browser was now reaching the backend but CORS headers were missing — the backend wasn't explicitly allowing the CloudFront origin.

Fix: Spring Security CORS configuration reading the allowed origin from an environment variable (FRONTEND_URL), so the same codebase works locally and in production without hardcoded URLs.
@Value("${frontend.url}")
private String frontendUrl;

@Bean
public CorsConfigurationSource corsConfiguration() {
    CorsConfiguration configuration = new CorsConfiguration();
    configuration.setAllowedOriginPatterns(
        List.of(frontendUrl, "http://localhost:3000")
    );
    configuration.setAllowedMethods(
        Arrays.asList("GET", "POST", "PUT", "DELETE", "OPTIONS")
    );
    configuration.setAllowCredentials(true);
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", configuration);
    return source;
}

What's Not Production-Grade

Being transparent about trade-offs matters. Here is what I would change with more time and a custom domain:

  • Three CloudFront distributions as HTTPS proxies is a workaround for not having a custom domain. In production: one domain, one ALB, one ACM certificate.
  • Skipping JWT issuer validation works but removes a security layer. The proper fix is making Keycloak's internal and external hostname identical.
  • Keycloak in start-dev mode is development configuration. Production Keycloak uses start mode with proper TLS.

I know what the upgrade path looks like. That's the point of this exercise.
