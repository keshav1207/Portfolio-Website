How I Built a Full CI/CD Pipeline as a Junior Dev (and Everything That Went Wrong)

By Keshav Callychurn

I recently built a complete CI/CD pipeline for my personal project RememberBirthdays using GitHub Actions. The project has two repos: a Spring Boot backend and a React frontend, each deploying to a different AWS target.

This is a full walkthrough of what I built, why I made each decision, and every problem I hit along the way. If you are setting this up for the first time, this will save you a few hours.

What I Built

The pipeline covers four phases:

  • Backend CI runs all 18 Maven tests on every push and pull request to main
  • Frontend CI runs all 16 React/Jest tests on every push
  • Frontend CD builds the React app, syncs to S3, and invalidates the CloudFront cache
  • Backend CD builds a Docker image, pushes to AWS ECR, SSHes into EC2, and restarts the container with Docker Compose

The deploy jobs use the needs keyword to gate on passing tests. A failing test blocks the deployment automatically.

Phase 0: Prerequisites

Before writing a single YAML file I set up the AWS infrastructure:

  • Created an IAM user (github-actions-deploy) with ECR, S3, and CloudFront permissions. Never use root credentials for automation.
  • Created a private ECR repository to store Docker images.
  • Attached an IAM role with AmazonEC2ContainerRegistryReadOnly to the EC2 instance so it can pull images at deploy time.
  • Stored all sensitive values as GitHub Secrets in each repo: AWS keys, SSH private key, EC2 host, bucket names, and CloudFront distribution ID.

This phase takes the longest but only needs to be done once.

The Problems I Actually Hit

This is the part tutorials skip.

1mvnw: Permission denied (exit code 126)

GitHub Actions runs on Linux. The mvnw wrapper script needs to be executable, but Windows does not track Unix file permissions. So the file existed in the repo but was not executable on the CI runner.

Fix: git update-index --chmod=+x mvnw, then commit. Git stores the executable bit and it carries over to any Linux environment.
2Tests pass locally, fail in CI

My React tests failed in CI with TypeError: expect(...).toBeInTheDocument is not a function. Locally all tests passed. The difference: setupTests.ts had never been committed to git. The file existed on my machine but git was silently not tracking it. Running git ls-files confirmed it was missing.

3Case sensitivity: setUpTests.ts vs setupTests.ts

Even after committing, CI still failed. Git had tracked the file as setUpTests.ts (capital U) because that is how I originally created it on Windows, which has a case-insensitive filesystem. On Linux, these are two different filenames and react-scripts looks specifically for setupTests.ts.

Fix: rename in two steps to force git to register the case change.
git mv setUpTests.ts setupTestsTemp.ts, then git mv setupTestsTemp.ts setupTests.ts.
4ESLint treating warnings as errors

The React build failed in CI with ESLint errors that did not fail locally. CI sets the CI=true environment variable, which causes react-scripts to treat all ESLint warnings as hard errors. I fixed the actual warnings rather than disabling CI mode: unused variables and a missing useEffect dependency.

5SSH connection timeout

The backend CD workflow timed out at the SSH step. The EC2 security group had no inbound rule for port 22 from GitHub Actions IP ranges.

Fix: Add an inbound rule allowing TCP port 22.
6CloudFront AccessDenied

The CloudFront invalidation step failed with AccessDenied. I had attached S3FullAccess and ECRFullAccess to the IAM user but forgot CloudFrontFullAccess.

Fix: Attach the policy in the AWS console and re-run the job.

What I Learned

  • CI surfaces environment differences invisible on Windows: file permissions, case sensitivity, missing commits.
  • The needs keyword gates deploys on passing tests. If tests fail, the deploy never runs.
  • IAM least privilege matters. The IAM user only has the exact permissions it needs.
  • npm ci over npm install in CI. It reads the lockfile exactly and gives you a reproducible build every time.
  • Every confusing error is a gap in mental model. Once you understand why something fails, the fix is always simple.

The End Result

Every push to main on either repo triggers a full automated pipeline. Tests run, build happens, deployment completes, all without touching the server manually. If a test fails, the deploy never runs.

For a junior developer, this is the kind of setup that shows you understand how real software ships, not just how to write features.
