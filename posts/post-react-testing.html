Adding React Testing Library to a CRA Project in 2025 — The Challenges Nobody Warns You About

ESM module conflicts, Keycloak auth mocking, and why testing from the user's perspective changes how you think about components

By Keshav Callychurn

I recently added 16 tests to the React frontend of my RememberBirthdays app. The app is built with React 19, TypeScript, Create React App, Keycloak OAuth2 authentication, axios, react-router-dom v7, MUI, and react-hook-form.

Every one of those libraries caused at least one problem during test setup. This article is the guide I wish I had before I started.

The Testing Stack

  • Jest — test runner, built into Create React App
  • React Testing Library — renders components and lets you query what the user sees
  • @testing-library/jest-dom — adds readable matchers like toBeInTheDocument()
  • @testing-library/user-event — simulates realistic user interactions

All of these were already installed in my package.json. The challenge was getting them to actually work together.

1ESM Module Conflicts

Jest runs in Node.js using CommonJS. Axios v1.x and react-router-dom v7 use ES modules internally. This mismatch caused three different errors before I could run a single test.

The fixes required in package.json:

  • transformIgnorePatterns — tell Jest to transform axios through Babel instead of skipping it
  • moduleNameMapper — explicitly point Jest to the CommonJS version of react-router-dom and react-router because their exports field confused Jest's resolver

And in setupTests.ts:

  • Polyfill TextEncoder and TextDecoder from Node's util module — jsdom does not include them but react-router-dom v7 requires them
None of these errors had obvious solutions from the error messages alone. Each one took research to understand.
2Mocking Keycloak Auth Context

Every component in my app uses AuthContext from react-oauth2-code-pkce to read the token and user data. In tests there is no Keycloak server — so every component would crash immediately trying to read from an empty context.

The solution was to use the real AuthContext.Provider with a fake value, wrapping each component in the render call:

<AuthContext.Provider value={{
    token: 'fake-token',
    tokenData: { realm_access: { roles: ['Admin'] }, sub: 'user-123' },
    logOut: jest.fn(),
} as any}>
    <Navbar />
</AuthContext.Provider>

The as any cast is needed because we only provide the fields our components actually use, not every field the real AuthContext requires. This is an acceptable trade-off in tests.

By passing different roles arrays I could test both admin and regular user views in the same test file — which is where the most interesting assertions live.

3Testing Async Data Loading

The AllBirthday page fetches data from the API inside a useEffect on mount. When the component first renders, the birthday list is empty. After the API call resolves, the list populates.

This means you cannot assert immediately after rendering — the data is not there yet. The waitFor utility from RTL solves this by retrying the assertion until it passes or times out:

(api.get as jest.Mock).mockResolvedValue({
    data: [{ id: 1, firstName: 'John', lastName: 'Doe' }]
});
renderAllBirthday();
await waitFor(() => {
    expect(screen.getByText('John')).toBeInTheDocument();
});

The RTL Philosophy — Test What the User Sees

The most important shift React Testing Library forces on you is this: stop testing implementation details, start testing behaviour.

Before RTL I would have tested things like: does the isAdmin state variable get set correctly? Does the birthdayData array have two items?

With RTL you test: does the Admin link appear on screen? Do both names appear in the table? This is a fundamentally better way to test because it reflects what actually matters — what the user experiences.

Key Lessons

  • Set up your test environment before writing a single test. Fighting ESM issues mid-way through a test is demoralising.
  • Create a renderX() helper for each component that wraps it with all required providers. It makes tests readable and reduces repetition.
  • Use queryByText (not getByText) when asserting something does NOT exist — getByText throws an error if the element is not found, which crashes the test before your assertion runs.
  • Mock at the right level. Mock the entire Navbar component inside other page tests — you are not testing Navbar there, so do not set up its dependencies.
  • userEvent is better than fireEvent for most interactions, but fireEvent.change is more reliable for date inputs in jsdom.

Conclusion

16 tests, all green. The setup took longer than the actual tests. But the result is a frontend that I can confidently change without worrying about breaking things silently.

If you are building a React portfolio project — adding even a handful of RTL tests signals to employers that you understand how production-grade frontend development works. The setup pain is worth it.
